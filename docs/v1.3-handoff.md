## Cross-computer deployment: no Bridge required

The design has no Bridge, proxy, local service, driver helper or Input runtime.
Both host paths use the keyboard's Raw HID interface directly:

```text
ChatGPT Desktop ── Channel 2 ──> KB16 firmware ──> Codex task/status lights
Mapper           ── Channel 3 ──> KB16 EEPROM configuration
```

The six Codex task lights, including white/blue/green/amber/red status,
breathing and host brightness, are sent directly by ChatGPT Desktop over
Channel 2. Mapper does not receive, proxy or translate those status messages.

To use the same functionality on another Windows x64 computer:

1. Install and sign in to ChatGPT Desktop.
2. Connect a KB16 already flashed with the matching compatibility firmware.
3. Copy the complete Mapper portable directory or complete ZIP if keymap or
   ordinary-layer-lighting editing is needed. Keep all seven published files
   together; do not copy only the EXE.

No .NET runtime, Input installation, Bridge package, QMK toolchain or
background process is required for normal use. The keyboard retains its saved
EEPROM configuration when moved between computers. Only firmware *building*
on another machine needs QMK plus the keymap and four files under
`firmware\qmk-core-patches`; only a new physical keyboard needs flashing.

For the v1.3.5 ordinary-layer HSV lighting controls, use the matching v1.3.5
firmware build. Earlier v1.3 firmware remains sufficient for the original
keymap editor, Channel 2 status lights and Codex native controls.

## Locked product model

- Four fixed layers: Codex, Number, Navigation and System. They cannot be added,
  removed or renamed.
- Codex layer contains exactly one each of:
  `AG00-AG05`, joystick up/right/left/down,
  `ACT06/07/08/09/12`, and Mic `ACT10`.
- `ACT11` is never emitted. ChatGPT groups Mic into `ACT10_ACT11`.
- Codex native controls may only be exchanged/reordered; duplicates and missing
  controls are invalid.
- Six AG status lights follow their AG controls when the controls move.
- Codex left encoder is locked to native `ENC_CC`, `ENC_CW`, `ENC_SW`.
- Codex right and middle encoders are editable.
- All three encoders are editable on the other three layers.
- Middle encoder release before three seconds executes its configured short
  action. A hold of about three seconds cancels the short action and returns to
  Codex.
- Ordinary-layer actions use the fixed four-byte descriptor
  `kind + modifiers + uint16 code` and support keyboard chords, media, mouse,
  layer controls, RGB, OLED brightness and guarded Bootloader entry.
- No macros, text entry, delayed sequences, program execution, Smart Actions,
  Work Louder Input integration or background service.

| Row 1 | AG00 | AG01 | Joystick up | Joystick right |
| Row 2 | AG02 | AG03 | Joystick left | Joystick down |

and encoder behavior.

## Firmware and transport

- USB identity remains `303A:8360`, usage `FF00:0001`, Report ID 6 and 64-byte
  reports.
- Codex JSON-RPC remains on Channel 2.
- Mapper binary configuration protocol uses Channel 3, version 1.
- Channel 3 header contains report ID, channel, version, opcode, 16-bit request
  ID, fragment index/count and payload length. Maximum payload is 55 bytes.
- Supported operations: Hello, read configuration, write begin, ordered chunk
  writes, commit and restore v1.2 defaults.
- Configuration payload is 340 bytes. Golden-vector CRC32 is `D82701A6` for
  native values 0-15 followed by zero actions.
- EEPROM uses two 356-byte A/B slots with magic, schema, generation and CRC32.
  Only a completely validated inactive slot becomes active.
- Commit is rejected while input is held. Bad version, length, ordering, CRC,
  semantic configuration or duplicate commit is rejected.
- Lighting stays on PWM DMA1 Channel 6. OLED, Bootmagic and STM32duino
  Bootloader remain in place.

### Locked visual decisions

- Use warm white, graphite black, brand orange `#E96E24` and semantic green.
- Green is reserved for real connected/success states.
- `#E96E24` is reserved for selected priority and applying changes to the
  device. Do not spread it across secondary controls.
- The UI uses borderless Soft UI. Static 1 px outlines were explicitly rejected
  by the user.
- Containers remain comparatively flat. Raised and recessed effects are used
  only where they communicate physical interaction or input state.
- Components use same-color surfaces with a consistent upper-left light source,
  upper-left highlights and lower-right soft shadows.
- Keycaps, inputs, chips, ordinary buttons, modifier keys and dialogs do not use
  static outlines. Keyboard focus must remain visible through a deliberate
  focus treatment.
- Selected mapping and current primary operation retain the orange hierarchy.
  Error remains semantic red.
- Knobs follow the user's supplied pale Soft UI reference: a shallow recessed
  circular groove, a same-color round disk and a small circular dimple near the
  top. Do not reintroduce metallic texture, dirty gray rings, a dark pointer or
  a toothed/gear-like perimeter.
- Combination-key modifiers are neutral mini keycaps, not visible checkboxes.
  Unselected keys are slightly raised; selected keys use a neutral inset pressed
  state. They must not use `#E96E24`.
- Encoder rotation targets must remain independently selectable but visually
  flat: no outer card, no drop shadow and no `Counterclockwise` / `Clockwise` caption.
- Use warm light shadows rather than dark gray elevation. Current WPF values
  use beige-gray tints around `#B1A18C` to `#B9AA96`, blur roughly `10-22`,
  depth `2-3` and low opacity.
